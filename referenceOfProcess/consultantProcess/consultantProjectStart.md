### 顧問執行專案之流程與安排

此文件說明顧問承接到案子後的步驟以及人天時數的規劃，旨在於建立系統設定中的標準階段服務項目顧問服務的模板`FrontEnd/manager-back-site/src/views/system/project-template/SystemProjectTemplateSettings.vue`

---

## A、接到案子後的步驟以及人天時數的規劃

### 一、 接案後的初始步驟：合約與需求分析

1. **審閱合約與需求規格書 (RFP)**
   首先需詳細檢查合約書與需求規格書，找出是否有**額外要求（即「額外長出的腳」）**或潛在陷阱 。例如，需特別留意驗收條件、交付期限（如 180 日內完工）以及特定的技術要求（如防護基準、風險報告）。

2. **標示重點**
   建議將會影響驗收的重點項目以**顏色標示**，包含專案目標（如建置 ISMS）、實施範圍（如特定局處）、交付文件格式（如紙本與電子檔各兩份）等 。

3. **確認行政與保密需求**
   在收到訂單後的三個工作天內，需與承辦人確認是否需要發文，並提供保密同意書給相關成員簽署 。

### 二、 撰寫計畫書與準備起始會議

1. **產出專案文件**
   承辦顧問需撰寫專案工作計畫書與起始會議簡報 。計畫書內應包含專案組織成員資料、品質管理及合約要求的各項細節。

2. **定案交付時程**
   依據合約推算精確的日期（非僅標示「30 天內」），並與客戶確認發文方式（是一整包發文還是逐項發文） 。

### 三、 人天時數與行程規劃 (預定行程表)

1. **編製人天規劃表（形式曆）**
   這份表是給客戶看的，旨在展現對全局的掌握 。

- **排序邏輯**：先將合約死扣的交付期限（如 30 天、120 天、180 天）定下來，再依時間軸插入起始會議、文件討論、資產盤點、風險評鑑、內稽與管審會議等活動 。

- **預留前置時間**：若涉及第三方法驗證，文件發行必須排在驗證前的三個月；所有交付文件應於期限前 10 個工作天提供給主管審核 。

2. **人天數計算邏輯 (Onsite/Offsite)**

- **標準比例**：原則上到場輔導（Onsite, S）半天或一天，對應的場外準備與後續處理（Offsite, O）通常設為 0.5 人天 。

- **特殊調整**：若專案複雜度高，需要更多時間準備資料（如超過 1.5 天），需先與主管溝通確認 。

- **多人參與**：若有三人共同出席起始會議（0.5 天），則計算為 1.5 人天 。

### 四、 後續落實與風險控管

1. **工作規劃執行紀錄表 (Excel)**
   每月需更新 Excel 表與形式曆，記錄實際投入的人天與進度，並於月會前提交給相關主管 。

2. **會議紀錄要求**
   每次到點後需產出會議紀錄，重點標註「本次/下次輔導重點」以及顧問與客戶各自的待辦事項（可用顏色區分） 。

## B、Onsite 與 Offsite 人天比例計算邏輯

根據提供的來源內容，計算 Onsite（到場輔導）與 Offsite（場外準備）的人天比例有其標準邏輯與調整機制，詳細說明如下：

### 1. 標準計算邏輯

在一般情況下，不論到場時間長短，場外準備的人天數通常設為固定值：

- **標準比例**：原則上不論到場（Onsite）是一天或半天，其對應的場外準備（Offsite）人天通常皆設定為 0.5。
- **換算示例**：
- 到場 1 天：比例為 1 : 0.5。
- 到場 0.5 天：比例為 0.5 : 0.5。

### 2. 多人參與的計算方式

若一個行程有多位成員共同出席，人天數會依人數倍增，但場外比例的原則基本不變：

- **計算公式**：人數 × 到場時間 = 總 Onsite 人天。
- **範例**：若起始會議（0.5 天）有 3 位成員參加，則計算為 1.5 人天（0.5 × 3）。
- **Offsite 分配**：場外人天通常會歸給主要負責的成員，因為難以精確判斷每個人投入的準備時間。

### 3. Offsite 的實質內容與必要性

場外人天（Offsite）並非虛報，而是包含以下必要工作：

- **前置準備**：去客戶端之前需要準備資料、分析現況。
- **後續處理**：回來後需根據現場討論結果調整文件、處理會議紀錄。
- **複雜任務**：例如「文件討論」階段，通常需要花費大量時間進行分析與調整，因此分配 Offsite 時數是合理的。

### 4. 特殊狀況的調整機制

如果專案具體情況特殊，比例可以調整，但需經過溝通：

- **高複雜度需求**：若客戶要求較多，或顧問需要花費超過 1 天或 1.5 天來準備資料，必須先與主管溝通並獲得確認後，才能調整人天比例。
- **低投入需求**：若僅是單純的教育訓練且教材已現成，或僅是去旁聽（觀摩）而非主導，則可能不產生或不變動 Offsite 時數。

> **總結來說**
> Onsite 與 Offsite 的規劃就像是「舞台演出」與「幕後排練」的關係。不論正式演出的時間（Onsite）是長是短，幕後的準備與收拾（Offsite）通常都有一套標準的基本功（0.5 人天）要完成，以確保演出的品質與成果符合預期。

## C、涉及第三方法驗證的特殊規劃

若專案涉及**第三方法驗證**（即外部稽核驗證），在進行人天與時程規劃時，有以下幾個關鍵的特殊細節需要留意：

### 1. 文件發行的「三個月」規則

這是規劃中最核心的限制。來源強調，若專案有第三方法驗證需求，**文件發行（發佈）的時間點必須設定在驗證日期的三個月之前**。這是為了確保管理體系在驗證前已經有足夠的運行紀錄。

### 2. 採行「逆推排程法」

在編製預定行程表（形式曆）時，必須以**驗證日期**作為終點站往回推算：

- **範例**：若預計驗證時間排在 5 月 4 日，則文件發行最晚必須排在 2 月初。
- **申請作業**：除了文件發行，還必須在行程中明確插入一個動作項：「**第三方法驗證的申請**」。

### 3. 增加特定工作階段

相較於不需驗證的單純輔導案（通常較單純，僅需兩頁規劃），涉及驗證的案子通常會多出關於「**外部稽核準備與驗證**」的規劃內容。這包含：

- **預留外部稽核申請時間**：需確認機關何時提出申請。
- **外部稽核後的矯正**：需考慮驗證後的缺失改進（如去年的矯正紀錄）是否會影響當年度的規劃。

### 4. 風險控管與月會追蹤

在人天執行紀錄與月會中，針對有驗證需求的案子，顧問需要特別識別並報告以下風險：

- **外部驗證申請進度**：是否已如期提交申請。
- **文件發行阻礙**：如果資產盤點或文件討論進度落後，導致無法如期發行文件，將直接威脅到三個月後的驗證時程。

### 5. 管理審查會議（管審會）的參與

雖然顧問不一定每次會議都參加，但針對涉及驗證的案子，主管會依據**專案風險**來判斷是否親自出席管審會，以確保客戶的高層支持與驗證順利進行。

> **總結來說**
> 規劃涉及驗證的專案就像是「限時完成的拼圖」，你必須先鎖定最後一塊拼圖（驗證日期）的位置，然後精確地回報並安插前三個月的關鍵拼圖（文件發行），只要這個關鍵步調錯了，後續的驗證計畫就會全盤受阻。

## D、已對應到系統標準階段模板（顧問諮詢）

以下內容已套用至標準階段設定（`SystemProjectTemplateSettings.vue`）的顧問諮詢模板：

1. 合約與需求審閱
   - 產出：合約重點摘要、需求規格檢核清單
   - 預估時間：4 小時（Onsite 0 / Offsite 4）
   - 期限：專案起日 + 3 天

2. 行政與保密需求確認
   - 產出：保密同意書、發文方式確認
   - 預估時間：2 小時（Onsite 0 / Offsite 2）
   - 期限：專案起日 + 3 天

3. 計畫書與起始會議準備
   - 產出：專案工作計畫書、起始會議簡報
   - 預估時間：8 小時（Onsite 0 / Offsite 8）

4. 起始會議
   - 產出：起始會議紀錄、待辦清單
   - 預估時間：8 小時（Onsite 4 / Offsite 4）

5. 形式曆與人天規劃
   - 產出：形式曆、人天規劃表
   - 預估時間：8 小時（Onsite 0 / Offsite 8）

6. 文件建置與訪談盤點
   - 產出：訪談紀錄、資產盤點清單、文件草案
   - 預估時間：12 小時（Onsite 8 / Offsite 4）

7. 風險評鑑與改善建議
   - 產出：風險評鑑報告、改善建議摘要
   - 預估時間：8 小時（Onsite 4 / Offsite 4）

8. 內稽／管審會議支援
   - 產出：內稽紀錄、管審會議紀錄
   - 預估時間：6 小時（Onsite 4 / Offsite 2）

9. 結案準備與交付
   - 產出：交付清單、結案報告
   - 預估時間：4 小時（Onsite 0 / Offsite 4）
   - 期限：專案起日 + 180 天

10. 第三方驗證準備（如適用）
    - 產出：驗證申請、文件發行
    - 預估時間：6 小時（Onsite 0 / Offsite 6）
    - 期限：專案迄日 - 90 天
